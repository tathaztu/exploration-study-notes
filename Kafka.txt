        ‚óã logs

    ‚óã Open the conf directory for all 3 nodes > make following changes

    - Rename zoo_sample.cfg to zoo.cfg

    ‚óã Open the data directory

Kafka Topic

    ‚óã   Particular stream of data
    ‚óã   Identified by "Name"
    ‚óã   Any number
    ‚óã   Loosely comparable to a DB Table | there are no constraints
    ‚óã   Accepts data without any verification
    ‚óã   Any message format - json, avro, text, binary, etc.
    ‚óã   Data Stream: Sequence of messages | Streaming through "topics"
    ‚óã   Cannot be queried. We use -
            -   Producers to send data
            -   Consumers to read data

Partitions

    ‚óã   Topic are split to Partition(s)
    ‚óã   Any number of Partitions
    ‚óã   Messages are "ordered" in a partition | Order of Insertion
    ‚óã   Offset: Incremental ID assigned to every message in a partition
    ‚óã   Data in a partition are immutable
    ‚óã   Data is retained in Kafla for a limited time | 1 week default
    ‚óã   Offsets can make sense only in conjunction with a partition
            -   Offset 1 Partition # 2 is not the same as Offset 1 Partition # 3
    ‚óã   Offsets are not reused (recycled) when the associated message gets
        deleted
    ‚óã   Ordering across partitions is not as straightforward as ordering within
        a partition
    ‚óã   Data is assigned randomly to a partition except when a key is provided


Producer
    ‚óã   Writes to Topics
    ‚óã   Decides the target Partition in advance
    ‚óã   Automatic recovery from broker/partitin failure
    ‚óã   Load Balancing
    ‚óã   Producers can optionally send a "key" with each message
            -   messages with a "key" are always stored in the same partition
            -   keys are typically used when we need to order messages
    ‚óã   Messages with "null" keys are stored on a round-robin basis


‚û§   Each Partition receives massages from one or more producers

Consumers
    ‚óã   Pull Model
    ‚óã   Know source broker
    ‚óã   May read from multiple partitions of the same topic
    ‚óã   Know how to recover on failure
    ‚óã   Data is read by sequentially by offset "within" a partition
    ‚óã   Needs to know the format/type expected by the Destination

‚û§   serialization and deserializaton type should not change during a topic
    lifecycle

Consumer Groups
    ‚óã   Idea of scaling
    ‚óã   Multiple consumers grouped together
    ‚óã   Within a consumer group, a partition can be assigned to only one
        consumer
    ‚óã   Multiple consumer groups on the same topic is allowed

Consumer Offsets
    ‚óã   Storing offsets that are read by a consumer (group)
    ‚óã   Stored in a Kafka topic called __consumer_offsets
    ‚óã   Offset Commits
            -   At least once
                    ‚ñ´   Committed right after a message is processed
                    ‚ùó   Message read again if problem in processing
                    ‚ùó   Requires idempotency in design
            -   At most once
                    ‚ñ´   As soon as messages are received
                    ‚ùó   Won't be read again if there is a problem in processing
            -   Exactly once
                    ‚ñ´   For Kafka workflows, use the Transactional API
                    ‚ñ´   For external workflows, use an idempotent consumer

Brokers
    ‚óã   Identified by ID
    ‚óã   Each broker contains certain Topic partitions
    ‚óã   Each broker has metadata about all other brokers, topics & partitions
    ‚óã   Multiple brokers form a Kafka cluster
    ‚óã   Connect to any broker (bootstrap-server) to connect to the cluster
			-	Recieves 
					‚ñ´   Connection + Metadata
					‚ñ´   List of all Brokers
    ‚óã   3 brokers is a good number to start with

Replication
    ‚óã   Production topics should have a replication factor > 1
    ‚óã   A replication factor of n can withstand the broker loss of (n - 1)
        replicas
    ‚óã   Partition Leader
            -   Only one broker can be the leader for a partition
            -   Producers can only send data to the leader
            -   Brokers, other than the leader, replicate the data from the
                leader
            -   If the Brokers, other than the Leader, are in sync with the
                leader, w.r.t. data, then they are called In-Sync-Replicas
            -   Consumers can only read from the Leader
                    **  there are some exceptions nowadays, starting from 2.4+,
                        where the consumers can read from the closest replica

Producer Acknowledgements

    Data write acknowledgement values

    ‚óã   acks = 0
            -   Producer won't wait for the leader's acknowledgement
            -   Possibility of data loss
    ‚óã   acks = 1
            -   Producer will wait for the leader's acknowledgement
            -   Limited data loss
    ‚óã   acks = all
            -   Producer will wait for the leader's & replica's acknowledgement
            -   No data loss

Zookeeper
    ‚óã   Legacy metadata system
            -   List of Brokers
            -   Broker Leader election
            -   Monitor and Inform Kafka about new topic, broker
                online/offline, topic delete, etc.
    ‚óã   Depricated and optional from 3.x
    ‚óã   Will not be supported from 4.x
    ‚óã   Has one leader(writes) and the rest are followers (reads)
    ‚óã   ZooKeeper by design, operates with an odd number of servers ‚îÄ
        1, 3, 5, 7...
    ‚óã   Does not store consumer offset data since v. 0.10

    From the slides:

    Should you use Zookeeper?

        Q.  With Kafka Brokers?
        A.  Yes, until Kafka 4.0 is out while waiting for Kafka without
            Zookeeper to be production-ready

        Q.  With Kafka Clients?
        A.  Over time, the Kafka clients and CLI have been migrated to leverage
            the brokers as a connection endpoint instead of Zookeeper

            Since Kafka 0.10, consumers store offset in Kafka and Zookeeper and
            must not connect to Zookeeper as it is deprecated

            Since Kafka 2.2, the kafka-topics.sh CLI command references Kafka
            brokers and not Zookeeper for topic management (creation, deletion,
            etc ... ) and the Zookeeper CLI argument is deprecated.

            All the APIs and commands that were previously leveraging Zookeeper
            are migrated to use Kafka instead, so that when clusters are
            migrated to be without Zookeeper, the change is invisible to
            clients.

            Zookeeper is also less secure than Kafka, and therefore Zookeeper
            ports should only be opened to allow traffic from Kafka brokers,
            and not Kafka clients

            Therefore, to be a great modern-day Kafka developer, never ever use
            Zookeeper as a configuration in your Kafka clients, and other
            programs that connect to Kafka.


    About Kafka KRaft

        ‚óã   In 2020, the Apache Kafka project started to work to remove the
            Zookeeper dependency from it (KIP-500)

        ‚óã   Zookeeper shows scaling issues when Kafka clusters have > 100,000
            partitions

        ‚óã   By removing Zookeeper, Apache Kafka can
                -   Scale to millions of partitions, and becomes easier to
                    maintain and set-up
                -   Improve stability, makes it easier to monitor, support and
                    administer
                -   Single security model for the whole system
                -   Single process to start with Kafka
                -   Faster controller shutdown and recovery time

        ‚óã   Kafka 3.X now implements the Raft protocol (KRaft) in order to
            replace Zookeeper
                -   Production ready since Kafka 3.3.1 (KIP-833)
                -   Kafka 4.0 will be released only with KRaft (no Zookeeper)
